# 发票比对功能说明与表设计

## 一、需求与目标

### 1.1 业务目标

以**系统当月数据为基准**，参考**供应商/财务提供的发票数据**进行比对，达到：

1. **发现漏开/少开发票**  
   系统有销售记录但发票未开或开票金额/数量少于系统，便于跟进供应商补开或核对原因。

2. **发现数据不一致**  
   系统与开票在金额、数量上存在差异（时间口径、品类口径、统计口径等），需**明确指出来**供人工核对，该合并的合并（如品类名称差异、同一税收分类下多行合并展示），不能合并的单独标为「不一致」。

3. **占比情况**  
   各维度下开票占系统比例、未开票/少开金额占比，便于从结构上把握差异。

4. **实际税负测算（增值税）**  
   均为增值税场景下，需要知道：  
   - 按系统销售额与适用税率推算的**应有税负**；  
   - 发票已体现的**已开票税负**；  
   - 二者差额（**税负缺口**）及占应有税负的比例。  
   据此了解「实际税务会是什么样子」，便于财务/税务管理。

### 1.2 功能定位与权限

- **入口**：与「数据导入」同级的**独立功能 Tab/页面**（如导航栏「数据导入 | 发票比对」），不嵌入数据导入页内部，便于职责清晰。
- **权限**：**仅对指定人员开放**。与数据导入权限分离，通过单独的环境变量配置可访问人员（如 `HTMA_INVOICE_COMPARE_ALLOWED_FEISHU_OPEN_IDS`），未配置人员不显示该入口且访问接口返回 403。超级管理员/HTMA_ADMIN 可继续拥有所有模块权限。
- **不急于开发**：先完善设计，再按步骤实现（建表 → 导入 → 比对逻辑 → 前端与占比/税负展示）。

## 二、现有发票 Excel 结构分析

### 2.1 已分析的 2 个文件（下载目录）

| 文件 | Sheet | 表头行 | 列说明 |
|------|--------|--------|--------|
| **2025年12月发票.xlsx** | Sheet1 | 第 2 行 | 三级类别名称、税收分类编码、税率、求和项:销售数量、求和项:开票金额 |
| **2026年1月发票.xlsx** | Sheet4 | 第 3 行 | 三级类别编码、三级类别名称、求和项:销售数量、求和项:开票金额、税率、税收分类编码 |

- 12 月：**无**「三级类别编码」，仅有「三级类别名称」；最后一行多为「总计」行（可过滤）。
- 1 月：**有**「三级类别编码」；最后一行为汇总行（开票金额合计，可过滤）。
- 两表均按**三级类别**（即系统里的**小类 / category_small**）汇总，维度一致，便于与 `t_htma_sale`、`t_htma_products` 按品类比对。

### 2.2 字段与系统对应关系

| 发票 Excel 列 | 含义 | 系统/比对对应 |
|---------------|------|----------------|
| 三级类别编码 | 小类编码（6 位如 101002） | `t_htma_sale.category_small_code` / `t_htma_products.category_small_code` |
| 三级类别名称 | 小类名称 | `t_htma_sale.category_small` 或 `category` / `t_htma_products.category_small` |
| 税收分类编码 | 税务编码（长数字） | `t_htma_tax_burden.tax_class_code` |
| 税率 | 如 0.13 | `t_htma_tax_burden.tax_rate`，用于算税负 |
| 求和项:销售数量 | 开票数量 | 与系统 `SUM(sale_qty)` 按品类比对 |
| 求和项:开票金额 | 开票金额（一般为含税） | 与系统 `SUM(sale_amount)` 按品类比对 |

- **税负**（税额）：  
  - 发票侧：`税额 = 开票金额 - 开票金额/(1+税率)` = `开票金额 * 税率 / (1+税率)`  
  - 系统侧：可用 `t_htma_tax_burden` 按品类(code/name)取得税率，再按系统销售额推算税负，与发票税额比对。

## 三、MySQL 表设计

### 3.1 发票明细表：`t_htma_invoice_detail`

用于存储**按月导入**的发票汇总数据（按三级类别一行一条）。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT UNSIGNED PK | 自增主键 |
| period_month | DATE | 账期月份（如 2025-12-01 表示 2025年12月） |
| category_small_code | VARCHAR(32) | 三级类别编码（可选，1月有、12月可空） |
| category_small_name | VARCHAR(128) | 三级类别名称 |
| tax_class_code | VARCHAR(32) | 税收分类编码 |
| tax_rate | DECIMAL(5,4) | 税率（如 0.13） |
| sale_qty | DECIMAL(14,2) | 开票销售数量 |
| invoice_amount | DECIMAL(14,2) | 开票金额（含税） |
| store_id | VARCHAR(32) | 门店，默认「沈阳超级仓」 |
| created_at / updated_at | DATETIME | 创建/更新时间 |

- **唯一约束**：同一月份、同一门店、同一三级类别（以名称或编码+名称为准）仅一条，便于幂等导入。
- **索引**：`period_month`、`category_small_name`、`category_small_code`，便于按月和品类汇总与 JOIN。

### 3.2 比对结果表（可选）：`t_htma_invoice_compare_result`

若希望**落库保存**每次比对结果，可增加此表；也可仅用 SQL 动态计算不落表。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT UNSIGNED PK | 自增主键 |
| period_month | DATE | 账期月份 |
| category_small_code | VARCHAR(32) | 三级类别编码（可空） |
| category_small_name | VARCHAR(128) | 三级类别名称 |
| system_sale_amount | DECIMAL(14,2) | 系统销售额 |
| system_sale_qty | DECIMAL(14,2) | 系统销售数量 |
| invoice_amount | DECIMAL(14,2) | 开票金额 |
| invoice_sale_qty | DECIMAL(14,2) | 开票数量 |
| amount_diff | DECIMAL(14,2) | 收入差额（系统 - 开票） |
| system_tax_amount | DECIMAL(14,2) | 系统推算税负（额） |
| invoice_tax_amount | DECIMAL(14,2) | 发票税负（额） |
| tax_diff | DECIMAL(14,2) | 税负差额 |
| store_id | VARCHAR(32) | 门店 |
| created_at | DATETIME | 比对时间 |

- 汇总时：按 `period_month` 汇总即可得到该月整体收入差额与税负差额。

## 四、与现有库表的关系

- **t_htma_sale**：按 `data_date` 落在当月、`store_id` 匹配，按 `category_small` 或 `category` 汇总得到系统销售额、销售数量。
- **t_htma_tax_burden**：按 `code`/`name` 与品类编码/名称对应，取 `tax_rate` 用于计算系统侧税负。
- **t_htma_products**：提供 `category_small_code`、`category_small`，用于发票「三级类别」与系统品类对齐（名称或编码映射）。

比对逻辑概要：按 `period_month` 取发票明细，按同月从 `t_htma_sale` 汇总系统数据，按品类（名称/编码）对齐后计算收入与税负差额；漏开/不一致/合并规则见下节。

## 五、比对逻辑细化（漏开 / 不一致 / 合并）

### 5.1 数据准备与口径

- **系统侧**：`t_htma_sale` 按 `data_date` 落在当月、`store_id` 匹配，按 **category_small**（或 category）汇总销售额、销售数量；税率来自 `t_htma_tax_burden` 按品类 code/name 匹配。
- **发票侧**：`t_htma_invoice_detail` 按 `period_month`、`store_id` 取当月按三级类别汇总的开票金额、开票数量、税率。
- **品类对齐**：以「三级类别名称」为主、「三级类别编码」为辅与系统品类匹配；若存在名称不一致（如发票「维生素」、系统「维生素类」），可通过**品类映射表**或配置合并为同一口径，便于「该合并的合并」。

### 5.2 漏开 / 少开

- **系统有、发票无**：某品类在系统当月有销售额，但发票明细中无该品类或该品类开票金额为 0 → 标为「可能漏开」，并给出系统销售额/数量。
- **系统多、发票少**：同一品类系统销售额（或数量）大于开票金额（或数量），差额标为「少开」，并算占比（少开金额/系统销售额）。

### 5.3 不一致

- **金额或数量差异**：系统与开票在金额或数量上不一致（含发票多开、系统少记等），超过设定阈值（如 1 元或 0.1%）时标为「不一致」，列表/导出中明确列出品类、系统数、开票数、差额，供人工核对。
- **时间/口径说明**：页面或导出中可注明「系统按自然日汇总、发票按开票日/结算日」等口径差异，便于解释部分差异。

### 5.4 合并展示

- **同品类多名称**：若配置了品类映射（发票名称 A、B 对应系统同一 category_small），汇总与比对时按合并后品类展示，避免重复行。
- **同税收分类多行**：若需按税收分类编码汇总展示，可将同一 tax_class_code 下多品类先合并再与系统按税率分组汇总对比。

## 六、占比与税负测算

### 6.1 占比指标（建议输出）

| 指标 | 含义 | 计算方式 |
|------|------|----------|
| 开票金额占比 | 当月开票金额占系统销售额比例 | 开票金额合计 / 系统销售额合计 |
| 未开票/少开金额占比 | 收入缺口占系统销售额比例 | (系统销售额 - 开票金额) / 系统销售额（可按整体或按品类） |
| 按品类占比 | 各品类开票额占该品类系统销售额 | 按品类分别计算，便于发现哪些品类漏开或一致 |
| 按税率占比 | 各税率档位下开票额、系统额及占比 | 按 tax_rate 分组汇总后算占比 |

以上均可做成汇总卡片 + 明细表，支持导出。

### 6.2 增值税下的实际税负测算

- **应有税负（系统口径）**  
  按系统当月销售额，结合各品类适用税率（来自 `t_htma_tax_burden`），计算：  
  `应纳税额 = 销售额(含税) × 税率 / (1+税率)` 或按不含税口径等价计算。汇总得到「若全部按系统销售开票，应缴纳的增值税额」。

- **已开票税负（发票口径）**  
  发票明细中：`已开票税额 = 开票金额 × 税率 / (1+税率)`，按当月汇总。

- **税负缺口与占比**  
  - 税负缺口 = 应有税负 − 已开票税负（若为负则表示开票多于系统口径，需关注原因）。  
  - 税负缺口占比 = 税负缺口 / 应有税负（或 已开票税负 / 应有税负 表示「已覆盖比例」）。

- **展示建议**  
  单独一块「实际税负测算」：应有税负、已开票税负、税负缺口、已覆盖比例；可选按税率拆分表格，便于财务判断实际税务情况。

## 七、功能入口与权限实现要点

### 7.1 入口

- **路由**：新增页面路由，如 `/invoice_compare`，与 `/import`、`/profit_share` 同级。
- **导航**：在主导航（与数据导入同一栏）增加「发票比对」链接；**仅对具备发票比对权限的用户展示**该链接，无权限用户不显示且直接访问该 URL 返回 403。

### 7.2 权限

- **模块名**：`invoice_compare`。
- **环境变量**：`HTMA_INVOICE_COMPARE_ALLOWED_FEISHU_OPEN_IDS`，逗号分隔的飞书 open_id；不配置则默认不放行（或与产品约定：不配置时是否等同于 import 权限）。
- **逻辑**：复用现有 `_has_module_access(module)` 机制，在 `env_map` 中增加 `"invoice_compare": "HTMA_INVOICE_COMPARE_ALLOWED_FEISHU_OPEN_IDS"`；超级管理员与 `HTMA_ADMIN_FEISHU_OPEN_IDS` 仍拥有所有模块权限。
- **前端**：页面加载时可选调用「当前用户权限」接口或根据 403 判断，无权限时提示「您暂无发票比对模块权限，请联系管理员」。

### 7.3 页面内容建议（与数据导入 Tab 并列）

- **Tab1：发票导入**  
  选择账期月份、上传当月发票 Excel（兼容 12 月/1 月两种格式），解析后写入 `t_htma_invoice_detail`，支持幂等（同月同品类覆盖）。
- **Tab2：比对结果与占比**  
  选择月份 → 展示：系统 vs 开票汇总、收入差额与占比、漏开/少开/不一致明细表（可合并展示）、按品类/按税率占比。
- **Tab3：实际税负测算**  
  选择月份 → 展示：应有税负、已开票税负、税负缺口、已覆盖比例；可选按税率拆分表。

## 八、后续步骤建议（不急于开发）

1. **建表**：执行 `scripts/24_create_invoice_tables.sql`，创建 `t_htma_invoice_detail`、`t_htma_invoice_compare_result`。
2. **权限与路由**：在 app 中增加 `invoice_compare` 模块与 `HTMA_INVOICE_COMPARE_ALLOWED_FEISHU_OPEN_IDS`，新增 `/invoice_compare` 路由及 403 逻辑；导航中按权限显示「发票比对」入口。
3. **导入**：实现发票 Excel 解析（兼容两种表头）、过滤总计行、写入 `t_htma_invoice_detail`（幂等）。
4. **比对与占比**：按月、按品类/税率汇总系统与发票，计算差额、漏开/少开/不一致标识、占比；可写 `t_htma_invoice_compare_result` 或动态计算。
5. **税负测算**：按上文公式计算应有税负、已开票税负、缺口与占比，单独区块展示。
6. **前端**：发票比对页（三块：导入 / 比对结果与占比 / 实际税负测算），仅指定人员可见可访问。

以上为在现有需求基础上的设计完善，开发前可再评审确认。

## 九、全量发票查询导出（税务平台 xlsx）

- **文件特征**：含 **「信息汇总表」**（商品明细行，含税收分类编码、货物名称、金额/税额/价税合计等）与 **「发票基础信息」**（按数电发票号码一行汇总）。
- **建表**：`scripts/26_full_invoice_raw_tables.sql` → `t_htma_full_invoice_import_batch`、`t_htma_full_invoice_header_raw`、`t_htma_full_invoice_line_raw`（启动看板时 `start_htma.sh` 会尝试自动执行）。
- **入口**：税务分析页 Tab **「开票台账」** → 上传 xlsx、选择账期；**同一门店同一账期重复导入会覆盖**该月全量数据。
- **与品类发票导入关系**：`t_htma_invoice_detail` 仍为**按三级品类汇总**的财务表；全量表保留**票面全字段**，用于明细追溯与 **品名归一化** 和 `t_htma_sale.product_name` 比对（「未开票/差异」区块）。
- **与税负测算**：仍使用「实际税负测算」Tab 及 `t_htma_tax_burden`；全量导入不替代税负逻辑。
