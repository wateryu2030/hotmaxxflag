# 消费洞察走势图逻辑分析

## 一、数据流

```
前端 loadConsumerInsightTrend(granularity)
  → fetch /api/consumer_insight_trend?granularity=week&days=90&period=...&start_date=...&end_date=...
  → 后端 api_consumer_insight_trend()
  → 返回 { labels, series: {total_sale, total_profit}, by_category, by_distribution }
```

## 二、后端 api_consumer_insight_trend 逻辑

### 1. 主走势（销售额柱 + 毛利额线）
- **数据来源**：主查询 `t_htma_sale`，按日/周/月 GROUP BY
- **条件**：无，始终执行
- **输出**：`labels`（x 轴）、`series.total_sale`、`series.total_profit`

### 2. 品类 Top5 销售额走势
- **数据来源**：`by_category`，结构为 `{ "服装": [v1,v2,...], "鞋子": [...], ... }`
- **构建条件**：`if rows and len(rows) <= 60` 才进入
- **逻辑**：取 Top5 品类，为每个品类查按日/周/月的销售额序列
- **问题**：当 `len(rows) > 60` 时，**by_category 为空**，品类 Top5 无数据
  - 例如：granularity=day，自定义 76 天 → 76 行 > 60 → 不构建

### 3. 经销方式走势（by_distribution）
- **数据来源**：`t_htma_sale` LEFT JOIN `t_htma_product_master`，使用 `m.distribution_mode`
- **构建条件**：`rows <= 60` 且 `_has_dist_mode`（information_schema 检测列存在）
- **问题**：若 `t_htma_product_master` 无 `distribution_mode` 列 → MySQL 1054 错误 → 500
- **当前防护**：`_has_dist_mode` 检测 + try/except，但生产环境可能未部署或检测失效

## 三、前端渲染逻辑

```javascript
if (td.error) {
  trendEl.innerHTML = '<div class="error">' + td.error + '</div>';
  catTrendEl.innerHTML = '';
  return;  // 两个图都不渲染
}
// 主走势：需要 labels.length
// 品类 Top5：需要 labels.length && Object.keys(by_category).length
```

**结论**：一旦 API 返回 500（td.error 有值），主走势和品类 Top5 都会显示错误或空白。

## 四、根因归纳

| 现象 | 根因 |
|------|------|
| 销售额与毛利额走势报错 | distribution_mode 查询抛错 → 500 → 前端显示 td.error |
| 品类 Top5 没数据 | ① API 500 时整个响应失败；② 或 len(rows)>60 时 by_category 未构建 |

## 五、修复策略

1. **彻底移除 distribution 块**：不查 distribution_mode，保证永不 500
2. **放宽 rows 限制**：by_category 改为 `len(rows) <= 120`，让更多场景有品类 Top5
3. **分离 by_category 与 by_distribution**：by_category 不依赖 product_master，可独立构建
