# 人力分析 Tab 设计文档

## 一、目标与范围

在好特卖看板中**单独新增 Tab「人力分析」**，用于将现有人力数据与销售数据结合，做更细致的效果分析，主要满足：

1. **经营 vs 管理拆分**：将人力成本分为「经营人力成本」「管理人力成本」，并通过**销售报表类目与人力岗位的手工映射**明确对应关系。
2. **人效计算**：提供总成本与销售的匹配效果、与人数匹配效果，看清销售/经营成果与人的关系。
3. **自定义时间段**：支持用户选择任意日期区间（如 12月25日–1月5日），系统将**按月统计的人力与销售数据按天数比例自动拆解**到该区间。
4. **按类目匹配人力**：选择时间段后，能展示各销售类目对应的人力成本汇总（组长+全职+兼职等合并）及明细。

**类目–人力映射由人工完成**：系统负责列出「销售日报的大类」与「人力成本中的岗位/类目」，由管理员在配置页完成两项决策：(1) 哪些人力成本归到**经营**、哪些归到**管理**；(2) 每个**销售大类**与哪些**经营人力岗位**一一对应。实现与联调通过 OpenClaw 自动化执行并检查。

## 二、经营人力成本 vs 管理人力成本

### 2.1 定义

| 类型             | 说明 |
|------------------|------|
| **经营人力成本** | 直接对应销售类目、可归入具体品类（如奢品、男装、潮运、户外、食品等）的人力。 |
| **管理人力成本** | 不直接对应单一销售类目，支撑整体运营的职能人力，如：物流、流量、客服、收银等。 |

### 2.2 层级约定（已确认）

- **只下沉一级**：  
  - **经营人力成本**：下面对应具体销售大类（手工匹配的类目），不再往下拆。  
  - **管理人力成本**：只到「流量、物流、客服、收银」等岗位一级；管理块内按岗位拆开展示，并可下沉到具体人名。

- **销售类目层级**：仅使用**大类**（`t_htma_sale.category_large`）。系统提供所有大类列表，由管理员手工匹配到经营人力成本映射表中，确保对应关系一致。

### 2.3 数据来源与对应关系

- **销售侧**：从 `t_htma_sale` 取 `category_large_code`、`category_large` 去重，得到**所有销售大类（代码+名称）**，与看板「大类」下拉、附图中一致，供配置页「拉取所有销售大类」展示与匹配。
- **人力侧**：**拉取所有人力成本汇总，以组长表为准**——从 `t_htma_labor_cost` 取去重后的 `position_name`（与薪资表、组长表岗位一致），展示时按岗位类型**组长(leader) 优先**排序，供配置页「拉取所有人力成本汇总（以组长表为准）」与新增映射时「人力岗位名」下拉选择。

### 2.4 类目–人力映射：人工配置的两项决策

映射关系**完全由人工在配置页完成**，系统只负责展示列表与持久化配置：

| 决策 | 说明 | 系统提供的列表 | 人工操作 |
|------|------|----------------|----------|
| **① 经营 vs 管理归属** | 每条人力岗位归入「经营」或「管理」 | 人力成本中的岗位名（来自 `t_htma_labor_cost.position_name` 或配置区已用岗位） | 新增映射时选择类型 `operational`（经营）或 `management`（管理）；未在映射表中的岗位在「无映射时」可默认全部归经营（见实现约定） |
| **② 销售大类 ↔ 经营人力** | 哪个销售大类对应哪些经营人力岗位 | 销售日报的大类代码+大类名称（来自 `t_htma_sale`）；人力岗位名 | 为每个经营大类配置一条或多条映射：`sales_category` = 大类名称，`labor_position_name` = 岗位名，`cost_type=operational`；匹配方式可选 exact/prefix |

- **经营**：配置页列出**所有销售大类**（大类代码+大类名称），管理员为需要计入「经营」的大类逐条配置对应的人力岗位名（如奢品→奢品组长、奢品全职、奢品兼职），每条映射可指定匹配方式（精确/前缀）与生效日期。
- **管理**：将「物流、流量、客服、收银」等岗位归入管理，每条对应一个岗位名、`cost_type=management`，`sales_category` 可为空或固定如「管理」；管理只到岗位一级，展示时按岗位拆分并可下沉到人名。

### 2.5 配置表设计（含生效日期）

**表：`t_htma_labor_category_mapping`**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 主键 |
| sales_category | VARCHAR(64) | 销售大类目（经营时填大类名；管理时可为空或固定如「管理」） |
| cost_type | VARCHAR(16) | `operational`=经营 / `management`=管理 |
| labor_position_name | VARCHAR(64) | 人力岗位名（与 position_name 匹配） |
| match_type | VARCHAR(16) | `exact` / `prefix` |
| effective_from | DATE | 生效开始日期（可选，NULL 表示不限） |
| effective_to | DATE | 生效结束日期（可选，NULL 表示不限） |
| sort_order | INT | 展示顺序 |
| created_at / updated_at | DATETIME | 审计 |

- 查询映射时按报表月份或时间段落在 `[effective_from, effective_to]` 内过滤（NULL 视为无界）。  
- 经营：每条记录对应「某销售大类 ↔ 某人力岗位」；管理：每条对应「管理 ↔ 某岗位名」，管理只到岗位一级。

## 三、人效计算设计

### 3.1 指标定义

在选定时间段、选定范围（全店或某类目）下：

| 指标 | 含义 | 计算方式 |
|------|------|----------|
| **总人力成本** | 该时间段内分摊得到的人力成本合计 | 见第四节「时间段拆解」 |
| **总人数** | 该时间段内参与统计的人数（全量） | 按月分摊后按人汇总：**全量统计、不去重**（成本已发生，按分摊到该时段的人次计） |
| **总销售额** | 该时间段内销售额 | 直接从 t_htma_sale 按 data_date 汇总 |
| **总毛利** | 该时间段内毛利 | 直接从 t_htma_sale 按 data_date 汇总 |
| **人效（成本）** | 单位人力成本带来的销售/毛利 | 销售额 / 总人力成本、毛利 / 总人力成本 |
| **人效（人数）** | 人均销售/毛利 | 销售额 / 总人数、毛利 / 总人数 |
| **投入产出比** | 产出/投入 | 毛利 / 总人力成本（或销售额 / 总人力成本，可配置） |

### 3.2 分层展示

- **全店维度**：总人力成本（经营+管理）、总销售、总毛利、总人数，以及上述人效与投入产出比。
- **经营类目维度**：按每个经营类目展示该类目下分摊的人力成本、人数、该类目销售额与毛利，及该类目的人效与投入产出比。
- **管理**：单独展示管理人力成本与人数，可不绑定销售类目，或与全店销售做整体人效。

这样可同时满足「总成本与销售的匹配效果」和「与人数的匹配效果」。

## 四、自定义时间段与按月数据拆解

### 4.1 需求

用户选择一段日期，例如 **12月25日–1月5日**，该区间横跨 12 月与 1 月。  
当前人力数据为**按月**统计（`t_htma_labor_cost.report_month`），销售数据为**按日**（`t_htma_sale.data_date`）。  
需要：

- **销售**：直接按 `data_date IN (start_date, end_date)` 汇总即可。
- **人力**：将涉及月份的「月总成本」与「月总人数」按**该月在本时间段内的天数占比**分摊到本时间段。

### 4.2 时间段内「月份」的拆解规则

设用户选择 `[start_date, end_date]`，例如 `2024-12-25` 到 `2025-01-05`。

1. **找出涉及的月份**：如 2024-12、2025-01。
2. **对每个月份 M**：
   - 计算 M 在本区间内的**天数**：  
     `days_in_range(M) = 区间内落在 M 的日期数`  
     例如 2024-12 在区间内有 7 天（12-25～12-31），2025-01 有 5 天（1-1～1-5）。
   - 该月总天数：`days_in_month(M)`（如 12 月=31，1 月=31）。
   - **权重**：`weight(M) = days_in_range(M) / days_in_month(M)`。
3. **人力成本分摊**：  
   某月总人力成本为 `C_month`，则该月对当前时间段的贡献：  
   `C_period_from_M = C_month * weight(M)`。  
   时间段总人力成本 = 各月 `C_period_from_M` 之和。
4. **人数处理（已确认）**：  
   **全量数据，不去重**。某月人数为 N，该月对时间段的贡献人数 = `N * weight(M)`；多个月份的贡献人数直接相加（同一人在多个月出现则按月分别计入），得到时间段内**总人次数**，与已发生的成本口径一致。

### 4.3 实现要点

- 人力按**月**汇总后再按**天占当月比例**分摊，避免重复计算。
- 若某月无人力数据，该月对时间段的贡献为 0，仅销售有数也可展示（人效中该月人力为 0 可特殊提示）。

## 五、按类目匹配的人力成本与明细

### 5.1 类目人力汇总

对每个**经营类目**（如奢品、男装、潮运、户外、食品等）：

- 通过 `t_htma_labor_category_mapping` 找到其对应的 `labor_position_name`（及 match_type）。
- 从 `t_htma_labor_cost` 中取出 `position_name` 匹配的**组长、全职、兼职**等记录（同一类目下组长+全职+兼职合并）。
- 先按**月**汇总该类目下人力成本与人数，再按第四节规则**按时间段分摊**，得到该类目在该时间段内的：
  - 总人力成本
  - 人数（或等效人·月/人数）
  - 明细：组长成本、全职成本、兼职成本等（可选：到岗位类型即可，或到人）。

### 5.2 明细展示要求

- **经营类目**：汇总行如「奢品：组长 + 奢品全职 + 奢品兼职 = 奢品类目人力成本合计」；可展开该类目下组长/全职/兼职分项成本与人数。
- **管理人力**：按岗位拆开展示（流量、物流、客服、收银等），每个岗位下可**下沉到具体人名**（展示该岗位在该时间段内涉及的人员列表及每人分摊成本）。

## 六、前端与交互（人力分析 Tab）

### 6.1 入口

- 在看板主导航中新增 Tab：**「人力分析」**（与「概览」「品类排行」「人力成本」等并列；若当前是独立页「人力成本」，可在其同层级增加「人力分析」Tab，或作为人力成本页内的子 Tab）。
- 仅在有权限时显示（与现有人力成本模块权限一致即可）。

### 6.2 主界面元素

1. **时间段选择器**  
   - 日期范围：开始日期、结束日期（必选）。  
   - 选择后自动触发：销售按日汇总、人力按第四节规则分摊。

2. **经营 vs 管理总览**  
   - 卡片或表格：经营人力成本、管理人力成本、总人力成本；总销售额、总毛利；总人数（或等效人数）。  
   - 人效：销售额/总人力成本、毛利/总人力成本、人均销售、人均毛利；投入产出比（毛利/总人力成本等）。

3. **经营类目明细表**  
   - 行：每个经营类目（来自销售大类目且已配置为经营）。  
   - 列：类目名、分摊人力成本、人数、该类目销售额、该类目毛利、人效（销售/成本、人均）、投入产出比。  
   - 支持点击展开：该类目下「组长 / 全职 / 兼职」等分项成本与人数。

4. **管理人力**  
   - 按岗位拆开展示（流量、物流、客服、收银等），每岗位：成本、人数、并可下沉到**具体人名**明细；整块汇总与全店销售的整体人效。

### 6.3 配置入口（类目–人力映射，人工完成）

- 提供**配置页**（人力分析页内「类目–人力映射配置」）：  
  - **拉取销售大类+中类**：与看板一致，使用 **`/api/categories`**（`level=large` 取大类，`level=mid&category_large_code=xxx` 取中类），返回 `[{code, name}]`；前端下拉 **value 用代码**，展示「代码 + 名称」；**匹配与存储均按大类代码、中类代码**，与 `t_htma_sale` 的 category_large_code、category_mid_code 对齐，避免按名称匹配导致中类拉取或匹配失败。  
  - **拉取人力成本汇总，以组长表为准**：从 `t_htma_labor_cost` 取去重岗位名，按组长(leader) 优先排序；供下拉与批量表使用。  
  - **批量映射（提高效率）**：表格行为人力岗位，列为类型、销售大类、销售中类（依大类级联）、匹配方式、生效起止；逐行选择后**批量保存**，避免逐条新增。  
  - **单条新增**：人力岗位名 → 类型 → 销售大类 → 销售中类（依大类级联）；经营时可选大类+中类，管理类不选。  
  - 映射表支持 **sales_category_mid**（销售中类名称，展示用）及 **sales_category_large_code、sales_category_mid_code**（与看板一致，按代码匹配）；每条可设生效日期范围。  
- 存储到 `t_htma_labor_category_mapping`。实现后通过 **OpenClaw 脚本**（`scripts/openclaw_labor_analysis_check.sh`）自动化检查。

## 十、已确认事项

1. **经营/管理只下沉一级**：经营下面对应具体类目（手工匹配）；管理只到流量、物流、客服、收银等岗位。  
2. **销售类目仅大类**：系统提供所有大类，用户手工匹配到经营人力成本表。  
3. **人数全量不去重**：按分摊到时间段的人次汇总，与已发生成本一致。  
4. **管理人力**：按岗位拆开展示，并可下沉到具体人名。  
5. **配置表**：增加 `effective_from` / `effective_to` 生效日期范围。  
6. **类目–人力映射由人工完成**：系统列出销售大类与人力岗位，人工决定 (1) 经营/管理归属，(2) 销售大类与经营人力岗位的映射。

按本设计实现后，通过 **OpenClaw 自动化执行并检查**：运行 `bash scripts/openclaw_labor_analysis_check.sh` 校验人力分析接口（categories、mapping、overview、by_category、management）与页面可访问性。

**部署与数据**：一键部署脚本 `deploy_htma_live.sh` **不会**执行 `13_create_labor_cost_table.sql`（该脚本含 DROP TABLE，会清空人力明细）；日常部署仅做加列与其它建表，历史人力数据保留。需要清空人力数据时，在「人力成本」页点击「清空人力数据」按钮（或调用 POST `/api/labor_cost_clear` 且 `confirm=yes`）手工触发。

实现时可在 `scripts/` 下新增 `16_create_labor_category_mapping.sql`，内容示例：
